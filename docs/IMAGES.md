# Image Handling

This document is the operational guide for images. The conceptual model
(`image` as a first-class entity, the dual spoiler filter) lives in
`/docs/DATA_MODEL.md` § Images. The R2 key convention is in
`/docs/ARCHITECTURE.md` § Deployment. The deferral of in-universe
documents is in ADR-011.

## Overview

- Images are **entities** of type `image`. They have ids, slugs, and
  carry their own properties (`url`, `license`, `attribution`,
  `spoiler_since`, `alt_text_key`, …).
- Other entities link to images via the **`depicted-by`** relation;
  the inverse direction (`depicts`) is auto-generated by the build
  pipeline.
- The same image can be linked from multiple entities — a group photo
  of the Straw Hats produces one image entity and nine `depicted-by`
  relations.
- Images carry licensing metadata per-entity; the wiki never serves an
  image that doesn't have `license` and `attribution`.

## Storage

Files live in **Cloudflare R2** under a flat namespace:

```
images/<image-slug>.<format>
```

Examples:

```
images/luffy-bounty-30m.webp
images/luffy-bounty-3b.webp
images/luffy-gear-5.webp
images/straw-hats-marineford-arrival.webp
images/gomu-gomu-no-mi.png
```

The layout is deliberately flat. Reuse is the default — an image with
several `depicted-by` targets has no single "owner" entity to nest
under. The slug carries the meaning.

**Public URL.** The `url` property on the image entity is the
CDN-fronted URL of the R2 object. Phase 4's upload server function
constructs this URL from the bucket configuration and the slug; do
not type it by hand in entity files.

**Deduplication.** The upload path computes a content hash on receipt;
if a file with the same hash already exists in R2, the existing slug is
reused and no new bytes are written. This avoids accidental duplicates
when contributors upload the same poster twice.

## Licensing policy

Every image carries `license` (a value from the `image-licenses`
vocabulary) and an `attribution` string. The combination determines
what the wiki may do with the file:

| Licence             | Hosting on R2 | Display surface      | Notes                                       |
| ------------------- | ------------- | -------------------- | ------------------------------------------- |
| `official-shueisha` | Yes           | All pages            | Attribution to Shueisha/Toei required.      |
| `fan-art-permitted` | Yes           | All pages            | Attribution must match artist's preference. |
| `fan-art-fair-use`  | Yes, small    | Thumbnail-size only  | No high-res; sparing use.                   |
| `screenshot-anime`  | Yes           | All pages            | Critical/documentary context required.      |
| `screenshot-manga`  | Yes, sparing  | Thumbnail or excerpt | Use only when prose can't substitute.       |
| `public-domain`     | Yes           | All pages            | No attribution required (still record one). |
| `cc-by`             | Yes           | All pages            | Attribution mandatory.                      |
| `cc-by-sa`          | Yes           | All pages            | Attribution mandatory; ShareAlike applies.  |

If a licence isn't on this list, the image MUST NOT be uploaded.
External hot-linking is forbidden — every served image lives in our R2
bucket so we control availability and SLA.

## Upload workflow

Phase 4 dashboard (Phase 1 has no UI; images entered by hand).

1. Editor opens an entity (e.g. `character:luffy`) and clicks
   `+ Image` on a `depicted-by` slot.
2. The form shows: file picker, `license` dropdown, `attribution`
   text input, optional `caption_key` + `alt_text_key` fields,
   `spoiler_since` source picker, role/period/context for the
   `depicted-by` relation.
3. On submit, the server function:
   - validates format and size,
   - hashes the file and checks R2 for an existing object,
   - uploads to `images/<image-slug>.<format>` (deriving the slug
     from the suggested name and falling back to a content-hash
     prefix on collision),
   - writes the `image` entity JSON and the `depicted-by` relation
     into the parent entity's JSON,
   - opens a PR labelled `images` carrying both files.
4. CI runs validation: every value-bearing field present, every
   reference resolves, licence is non-empty, attribution is
   non-empty.

## Spoiler handling

Two filters apply when deciding whether to render an image:

1. **`image.spoiler_since`** — is the image itself safe at the user's
   progression? A Gear 5 image with `spoiler_since:
   "manga-chapter:1044"` is hidden from anyone before that chapter,
   regardless of which entity it appears on.
2. **`depicted-by` qualifier `since`** — is _this_ depiction
   contextually accurate at the user's progression? Luffy's 3-billion
   wanted poster has `since: "manga-chapter:1053"` on its
   `depicted-by` relation; before chapter 1053, the depiction is
   filtered even though the image itself might have a later
   `spoiler_since`.

Read-path pseudocode:

```
visibleDepictions(entity, progression) =
  entity.relations
    .filter(r => r.type === "depicted-by")
    .filter(r => reachable(progression, r.qualifiers.since ?? r.target.spoiler_since))
    .filter(r => reachable(progression, r.target.spoiler_since))
```

`reachable(progression, source)` is the same source-reachability
function used elsewhere in the spoiler filter (see
`/docs/BUILD_PIPELINE.md` § 8).

If no image passes the filter for a `role: "primary_portrait"` slot,
the UI falls back to a generic silhouette or to an earlier-period
portrait that does pass — implementation choice for the apps, not a
data-model concern.

## Reuse across entities

A single image entity can be the target of many `depicted-by`
relations. The qualifier set captures how that image relates to each
depicted entity:

```json
// Inside crew:straw-hats.json
{
  "type": "depicted-by",
  "target": "image:straw-hats-marineford-arrival",
  "qualifiers": { "role": "group_photo", "since": "manga-chapter:567" }
}

// Inside character:luffy.json
{
  "type": "depicted-by",
  "target": "image:straw-hats-marineford-arrival",
  "qualifiers": { "role": "group_photo", "since": "manga-chapter:567" }
}

// …and seven more for the rest of the crew.
```

The build pipeline materialises the auto-inferred `depicts` direction
on the image entity: `depicts character:luffy`, `depicts
character:zoro`, …, `depicts crew:straw-hats`. The maintainer never
authors the inverse.

**When NOT to reuse.** If the depiction differs meaningfully — same
artwork but cropped, recoloured, captioned differently — author a
distinct image entity. The R2 dedup will collapse identical bytes; it
won't collapse "looks similar enough."

## Captions and accessibility

- **`alt_text_key`** is required on every image. The localised string
  it points to is the screen-reader description. It is _not_ the
  caption.
- **`caption_key`** is optional and historisable. The caption that
  appears under the image in prose context. Treated like any other
  localisable property: the key resolves through the standard i18n
  pipeline, locale-aware.
- Both keys follow the project's i18n key convention
  (`<entity-type>.<entity-slug>.<property>.<variant>`), see
  `/docs/I18N_STRATEGY.md`.

## Migration plan: images → documents

A future entity type `document` will model in-universe diegetic
objects (wanted posters, vivre cards, newspapers, letters, maps,
flags, manuscripts). It will carry richer fields than `image` does —
issuing entity, document subtype, in-universe authorship, validity
period — and enable queries like "all wanted posters issued by the
Marines".

**Migration is non-destructive.**

- Existing `image` entities representing such objects are kept as-is.
- A new `document` entity is created for each diegetic object
  (`document:luffy-wanted-poster-30m`, etc.).
- The `document` entity carries a `depicted-by` relation to its image
  (the image _depicts_ the document).
- Existing `depicted-by` relations from characters/crews to the
  poster-image are rewired to point at the new `document` entity if
  the depiction is "depicting the wanted poster", or kept if the
  depiction is "depicting Luffy".

See `/IDEAS.md` § "In-universe documents" for the long-form thinking
and ADR-011 for the deferral rationale.
